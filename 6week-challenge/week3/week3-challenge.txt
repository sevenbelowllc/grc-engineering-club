A policy you run on your laptop catches your own mistakes. A policy you run in CI catches everyone's, on every change, forever. This week you take the three policies from week 2 and wire them into a gate that runs on every pull request and blocks the ones that break a control.

This is the week GRC stops being a meeting. Nobody reviews the change for encryption. The pipeline does, in seconds, and a human only gets involved when something is actually wrong.

Starter code: grab `week-3-starter.zip`. It is a skeleton `grc-gate.yml` with the trigger and job set up and the real steps left as TODOs, plus a README that lays out the repo structure. You write the steps.

What you are building
A GitHub Actions workflow that, on every pull request to main:

Checks out the repo.

Installs Conftest at a pinned version.

Runs your three policy namespaces against your Terraform plan.

Fails the job if any policy reports a failure.

Uploads the results as an evidence artifact, whether the run passed or failed.

Then you prove it works by opening two pull requests: one that passes and merges, one that breaks a control and gets blocked.

The plan, the simple way

Your policies read a Terraform plan in JSON. The simplest free way to get one into CI is to generate it locally and commit it: `terraform show -json tfplan > plan.json`. The workflow runs Conftest against that committed file. No cloud credentials in CI, no secrets, nothing to pay for.

That is the path for this week. It keeps the focus on the gate mechanics, which is the thing you are actually learning. The production upgrade, where CI generates the plan itself, is the stretch goal below.

Three things that trip people up
Permissions. A workflow gets minimal permissions by default. To comment on a PR you need `pull-requests: write`. If you do the OIDC stretch, you also need `id-token: write`, and leaving it out makes OIDC fail silently. Set permissions explicitly.

Save evidence on failure. The natural mistake is to let a policy failure abort the job before the evidence uploads. That is backwards. The whole reason to capture CI evidence is to have a record of the failure. Mark the upload step to run always, not just on success.

Fail closed. The gate has to fail the job when a policy fails. Conftest exits non-zero on a violation, but if you swallow that exit code to capture the JSON output, you then have to make the pass or fail decision yourself from that JSON. Decide deliberately. A gate that captures violations but still passes the build is not a gate.

The two-PR demonstration

This is the actual deliverable, and it is what goes on your portfolio.

1. Green PR. Open a pull request with your compliant plan. The gate runs, every policy passes, the check goes green, the PR is mergeable.

2. Red PR. Open a pull request where the plan violates a control. Regenerate the plan from a workspace with the encryption block removed, or flip one public-access flag to false and regenerate. The gate runs, the matching policy fails, the check goes red.

Turn on branch protection for main and require the `grc-gate` check. Now the red PR cannot be merged, by anyone, until it is fixed. That sentence is the entire value proposition of this week. Screenshot both checks.

Cost and cleanup
Free. GitHub Actions covers this on the free tier, and the committed-plan path uses no cloud resources. Delete your test branches when you are done. The workflow file stays. It is the deliverable.

On GCP?

The gate is cloud-agnostic. GitHub Actions runs the same, Conftest runs the same, your policies match whatever cloud your plan describes. Nothing changes here.

Stretch: generate the plan in CI with OIDC

The real version does not commit a plan. CI generates it, which means CI needs to read your cloud. The right way to do that is GitHub OIDC: you create an identity provider and an IAM role that trusts tokens from your specific repository, and the workflow assumes that role with no stored keys.

The two things that matter are binding the trust to your exact repo (not a wildcard that every repo on GitHub could use) and giving the role only read access. If you build this, you have done something most GRC engineers have never set up, and it belongs on your portfolio with a paragraph explaining why keyless beats stored credentials.

Make it a portfolio piece
The `.github/workflows/grc-gate.yml` in your public repo.

Links to the two pull requests, one green, one red, visible in the repo's history.

A README section explaining what the gate enforces and what happens when a control breaks.

Post both checks on LinkedIn, green and red side by side. Tag GRC Engineering Club, use #GRCEngClubChallenge, and explain why a blocked merge is better than a caught mistake.

Common snags
OIDC token errors. The trust policy's subject condition must match your repo's token. Bind it to `repo:OWNER/REPO:*` and no looser.

Conftest finds no policies. The `--policy` path is relative to the step's working directory. Pass a path that resolves from where the step runs.

The gate passes when it should fail.* You captured the violations but did not act on them. Make the job exit non-zero when the results contain any failure.
